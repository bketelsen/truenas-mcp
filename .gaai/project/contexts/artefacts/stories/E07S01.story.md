# Story: Read-only flag prevents destructive tool registration

## User Story

**As a** NAS owner using the MCP server with AI assistants,
**I want** a `--read-only` flag on the `serve` command,
**so that** I can allow AI agents to query my NAS without any risk of them creating, deleting, or modifying resources.

---

## Context

The MCP server currently registers all tools — including destructive ones (dataset/snapshot/share create/delete, alert dismiss, app start/stop/restart). When connecting an AI assistant, there is no way to prevent it from invoking state-changing operations. A `--read-only` flag solves this by only registering read-only tools when enabled.

---

## Acceptance Criteria

- [ ] AC1: The `serve` command accepts a `--read-only` boolean flag (default: false), also settable via `TRUENAS_READ_ONLY` environment variable
- [ ] AC2: When `--read-only` is true, the MCP server does NOT register any tools that create, delete, update, dismiss, start, stop, or restart resources
- [ ] AC3: When `--read-only` is true, all read-only tools (list, get, inspect, info) remain registered and functional
- [ ] AC4: When `--read-only` is false or omitted, all tools are registered (no behavioral change from current state)
- [ ] AC5: The server logs or prints which mode it is running in at startup

> All acceptance criteria must be verifiable. Vague criteria are invalid.

---

## Out of Scope

- Per-tool allow/deny configuration
- Blocking calls at the handler level (tools are simply not registered)
- Changes to tool behavior or response format

---

## Dependencies

- E01S03 (MCP server initialization)

## Notes

Destructive tools to exclude in read-only mode:
- `truenas_dataset_create`, `truenas_dataset_delete`
- `truenas_snapshot_create`, `truenas_snapshot_delete`
- `truenas_smb_create`, `truenas_smb_delete`
- `truenas_nfs_create`, `truenas_nfs_delete`
- `truenas_alert_dismiss`
- `truenas_app_start`, `truenas_app_stop`, `truenas_app_restart`
